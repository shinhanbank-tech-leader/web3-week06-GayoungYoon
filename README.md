## 레포트 내용



**스마트컨트랙트는 은행 시스템을 통째로 대체하기보다는, 기존 장부형 시스템 위에 결제·정산·공유 상태 기록 기능을 추가하는 방향으로 활용될 가능성이 높다.**

은행의 핵심은 “돈을 이동시키는 기능”이 아니라 “거래를 책임 있게 관리하는 장부와 운영 구조”에 있기 때문이다. 반면 스마트컨트랙트의 핵심은 “사전에 정한 규칙을 공유된 환경에서 자동 실행하는 능력”에 있다. 스마트컨트랙트는 자동 실행, 투명한 상태 기록, 공유 원장 기반의 최종 결제 처리에는 강점이 있다. 반면 은행 시스템은 규제 대응, 계정 관리, 예외 처리, 정정, 취소, 고객 보호, 기관 간 책임 조정과 같은 운영 중심 업무에 강점이 있다. 즉 두 시스템은 같은 기능을 다른 방식으로 구현하는 경쟁 관계라기보다, 애초에 강한 영역이 다른 구조라고 보는 편이 정확하다.

따라서 금융 시스템의 미래는 어느 한쪽이 다른 한쪽을 완전히 대체하는 방향보다는, **장부형 시스템은 계속 유지하되 최종 결제 또는 특정 자산 이전 구간에 온체인 절차가 추가되는 방향**으로 보는 것이 합리적이다. 단순 송금, 자동 정산, 조건부 실행과 같은 일부 업무는 스마트컨트랙트가 대체할 수 있지만, 규제·예외 처리·정정·고객 보호를 포함하는 은행 코어 기능은 당분간 장부형 시스템이 중심일 가능성이 높다.

## 1. 스마트컨트랙트는 무엇을 대체할 수 있고, 무엇을 대체하기 어려운가

스마트컨트랙트는 규칙이 명확하고 예외가 적은 업무는 대체할 수 있지만, 규제·운영·보정이 핵심인 업무는 대체하기 어렵다.

스마트컨트랙트의 가장 큰 강점은 “사전에 정의된 규칙을 자동으로 실행하는 능력”이다. 예를 들어 조건부 지급, 단순 송금, 이자 분배, 만기 도래 시 자동 실행, 담보 청산과 같은 기능은 룰이 명확하고 상태 전이 조건이 비교적 단순하다. 이런 업무는 사람의 개입을 줄이고, 공동 원장 위에서 결과를 공유하며, 사후 검증 가능성도 높일 수 있기 때문에 스마트컨트랙트와 잘 맞는다.

반대로 은행 시스템에서 중요한 업무는 대부분 단순 규칙형이 아니다. 고객의 실명 확인, 이상거래 탐지, 자금세탁 방지, 세금 계산, 민원 대응, 지급정지, 사고 거래 복구, 계정 동결 및 해제, 정책 변경 반영 등은 모두 외부 제도와 운영 조직의 판단이 함께 개입된다. 또한 금융 상품은 시장 상황, 규제 변화, 내부 통제 기준에 따라 예외 규칙이 계속 추가되며, 실제 서비스 환경에서는 “원칙보다 예외가 더 중요한 경우”도 많다. 스마트컨트랙트는 이런 운영 판단을 자체적으로 수행할 수 없고, 외부 판단 결과를 받아 실행하는 보조 레이어에 가깝다.

따라서 스마트컨트랙트가 대체 가능한 영역과 어려운 영역은 아래와 같이 구분할 수 있다.

| 구분 | 스마트컨트랙트가 대체 가능한 영역 | 스마트컨트랙트가 대체하기 어려운 영역 |
|---|---|---|
| 업무 성격 | 규칙 기반 자동 실행 | 운영·규제·정책 기반 처리 |
| 대표 예시 | 단순 송금, 조건부 지급, 자동 정산, 이자/수수료 분배 | KYC/AML, 세금 처리, 계좌 관리, 민원 대응, 사고 정정 |
| 로직 특성 | deterministic, 사전 정의 가능 | 예외 다수, 외부 판단 필요 |
| 변경 빈도 | 비교적 낮음 | 높음 |
| 책임 주체 | 코드 중심 | 기관 중심 |
| 수정 가능성 | 낮음 | 높아야 함 |
| 운영 개입 필요성 | 낮음 | 매우 높음 |

실무적으로 더 분명하게 말하면, 스마트컨트랙트는 **“실행 규칙”**에는 강하지만 **“운영 가능성”**에는 약하다. 반대로 은행 시스템은 자동 실행성에서는 비효율이 있을 수 있어도, 운영 가능성과 책임 구조에서는 압도적으로 강하다. 이 차이가 결국 전체 대체가 어려운 이유다.

## 2. 왜 은행 시스템은 여전히 장부형 구조가 필요한가

은행 시스템은 단순 거래 처리 시스템이 아니라, 정정과 예외 대응이 가능한 장부형 시스템이어야 하기 때문에 기존 구조가 여전히 필요하다.

은행 업무의 핵심은 “거래를 실행했다”가 아니라 “거래 결과를 책임 있게 관리한다”는 데 있다. 실제 은행의 송금 기능을 보면, 사용자는 단순히 돈을 보내는 버튼만 누르지만 시스템 내부에서는 훨씬 많은 일이 일어난다. 잔액 검증, 한도 체크, 제재 대상 여부 조회, 거래 시간 조건, 수수료 계산, 상대 기관 연계, 실패 시 보류 처리, 거래 확정 후 회계 반영, 오류 시 정정까지 모두 장부형 시스템 위에서 처리된다.

장부형 시스템의 가장 큰 강점은 **상태를 유연하게 관리할 수 있다는 점**이다. 거래는 pending, approved, posted, failed, reversed, adjusted 등 다양한 상태를 가질 수 있고, 운영자는 이 상태를 기준으로 업무를 추적하고 보정할 수 있다. 또한 같은 거래라도 “요청 시점”, “승인 시점”, “실제 출금 시점”, “대외 정산 완료 시점”, “고객 통지 시점”이 분리될 수 있으며, 각각의 단계마다 실패·재시도·보류가 가능하다.

스마트컨트랙트는 이러한 다단계 운영 장부와는 철학이 다르다. 컨트랙트는 주어진 함수 호출에 대해 조건이 맞으면 상태를 바꾸고, 맞지 않으면 revert 한다. 물론 컨트랙트 내부에 복잡한 상태 머신을 설계할 수는 있지만, 은행이 운영 환경에서 필요로 하는 수준의 정정·보정·사후 운영 유연성을 확보하려면 결국 관리자 권한, 보류 상태, 정정 함수, 별도 오프체인 프로세스를 다시 넣어야 한다. 그렇게 되면 오히려 “순수 온체인 시스템”의 장점은 줄고, 전통 장부 시스템의 특성을 다시 구현하게 된다.

즉 은행 시스템이 장부형이어야 하는 이유는 기술 한계 때문이 아니라, **금융 서비스의 본질이 정정 가능성과 책임 가능성을 요구하기 때문**이다.

## 3. 전체 오프체인 구조와 일부 온체인 구조의 비교

현실적으로 가장 유력한 구조는, 기존 금융 시스템의 장부와 운영 체계는 유지하고 최종 결제 또는 기관 간 가치 이전 단계에만 온체인을 도입하는 하이브리드 구조이다.

전체 오프체인 구조에서는 고객 거래의 접수부터 검증, 내부 장부 반영, 기관 간 정산, 사후 정정까지 모두 내부 시스템과 기존 지급결제망에서 수행한다. 이 구조는 성숙했고, 예외 처리와 운영 보정이 가능하며, 제도 변경에 맞춰 유연하게 바꿀 수 있다. 다만 기관 간 공유 원장이 없고, 정산 단계가 분리되어 있어 처리 시간과 중간 조정 비용이 발생한다.

반면 일부 온체인 구조에서는 거래의 실질적 업무 판단은 여전히 은행 내부에서 수행하고, 온체인은 마지막 가치 이전 또는 최종 결제 기록에 활용한다. 예를 들어 내부 장부에서 출금 가능 상태를 만든 뒤, 그 금액에 해당하는 예금토큰 또는 스테이블코인을 발행하거나 이동시켜 상대 기관에 전달하고, 상대 기관은 다시 이를 내부 장부에 반영하는 방식이다. 이 구조에서는 은행의 운영 유연성과 블록체인의 최종성·공유 가능성을 동시에 활용할 수 있다.

이를 비교하면 다음과 같다.

| 항목 | 기존 전체 오프체인 구조 | 하이브리드 구조(일부 온체인) |
|---|---|---|
| 기본 장부 | 은행 내부 DB 중심 | 은행 내부 DB + 일부 온체인 기록 |
| 업무 판단 | 내부 시스템 | 내부 시스템 |
| 기관 간 가치 이전 | 기존 지급결제망 | 예금토큰/스테이블코인 등 온체인 이동 가능 |
| 정정 방식 | 직접 수정, 정정 분개, 재처리 가능 | 오프체인 보정 가능하나 온체인 기록은 원칙적으로 직접 삭제 불가 |
| rollback 개념 | DB transaction rollback 및 운영 rollback 가능 | 같은 tx 안에서는 revert 가능하나, 체인 확정 후에는 reverse transaction 필요 |
| 감사 및 추적 | 기관별 로그/장부 기반 | 장부 + 체인 기록 병행 |
| 유연성 | 매우 높음 | 중간 수준 |
| 최종성 | 정산 단계에 따라 지연 가능 | 온체인 구간은 빠른 finality 가능 |

## 4. Revert, rollback, reverse transaction의 차이

이 비교에서 가장 중요한 차이는 **revert와 rollback의 차이**다. 은행 시스템에서는 이미 수행된 거래라도 운영자가 장부를 수정하거나 정정 분개를 통해 실질적으로 원상 회복에 가까운 처리를 할 수 있다. 반면 스마트컨트랙트에서 revert는 **같은 트랜잭션 실행 중** 상태 변경을 무효화하는 개념일 뿐이다. 일단 트랜잭션이 블록에 포함되어 확정되면, 은행식 rollback처럼 과거 상태를 지우는 것은 불가능하고, 새로운 반대 거래를 통해 경제적 효과를 상쇄해야 한다. 즉, 은행 시스템의 정정 개념은 rollback과 adjustment 중심이고, 블록체인의 정정 개념은 revert와 reverse transaction 중심이다. 

은행 시스템에서 rollback은 주로 데이터베이스 트랜잭션 문맥에서 이해된다. 즉 하나의 업무 처리 중 오류가 나면 commit 이전 상태로 돌아갈 수 있고, commit 이후라도 운영자 정정, 취소 거래, 보정 분개 등을 통해 실질적으로 거래를 수정할 수 있다. 은행이 관리하는 장부는 중앙 집중적으로 통제되므로, “이미 처리된 거래”를 사후 운영으로 다시 다룰 수 있다는 점이 핵심이다.

반면 블록체인에서 revert는 컨트랙트 함수 실행 도중 조건이 맞지 않을 때 해당 트랜잭션 내의 상태 변경을 무효화하는 장치다. 즉 아직 체인에 반영되기 전의 실행 흐름을 취소하는 개념이다. 그러나 체인에 확정된 이후의 거래는 revert 대상이 아니다. 이때 필요한 것은 reverse transaction 또는 adjustment transaction이다. 즉 잘못된 송금이 있었다면 기존 거래를 삭제하는 것이 아니라, 그 거래를 참조하여 반대 효과를 내는 새 거래를 추가로 발생시켜야 한다.

이 차이는 실무적으로 매우 중요하다. 은행은 “과거 거래를 관리 가능한 기록”으로 보지만, 블록체인은 “과거 거래를 변경 불가능한 사실”로 본다. 따라서 스마트컨트랙트를 금융 서비스에 적용하려면 반드시 다음과 같은 설계가 필요하다.

- 원거래를 참조하는 정정 거래 구조
- pending / settled / reversed / adjusted 상태 분리
- 관리자 또는 운영자 권한 정의
- 온체인 구간 이전 단계에서 충분한 사전 검증 수행

즉 블록체인을 쓰더라도, 은행식 정정 요구사항을 만족하려면 결국 장부형 개념을 상당 부분 다시 도입해야 한다.

## 5. 가스, 저장공간, 함수 구조는 은행 시스템과 무엇이 다른가

 스마트컨트랙트는 비용이 드는 공개 상태 머신이고, 은행 시스템은 운영 가능한 장부형 애플리케이션이므로 가스, 저장공간, 함수 구조 모두 설계 철학이 다르다.

#### 5.1 가스

스마트컨트랙트에서는 연산 자체가 비용이다. 함수가 실행될 때 루프를 얼마나 도는지, storage를 몇 번 수정하는지, 외부 호출을 하는지에 따라 사용 가스가 달라지고, 이는 곧 사용자가 지불해야 할 수수료가 된다. 더 중요한 점은, 트랜잭션이 실패하더라도 이미 사용한 연산 비용은 되돌아오지 않는다는 것이다. 따라서 스마트컨트랙트는 단순히 “동작하느냐”가 아니라 “얼마나 저렴하게, 한도 내에서 동작하느냐”까지 고려해야 한다.

반면 은행 시스템에서는 연산 비용이 직접 거래 단위로 노출되지 않는다. 물론 CPU, 메모리, DB I/O, 네트워크 비용은 존재하지만, 이는 인프라 비용으로 흡수되며 함수 호출 단위로 사용자에게 직접 과금되지 않는다. 개발 관점에서도 은행은 주로 TPS, 응답 시간, 배치 처리량, DB 부하를 본다. 스마트컨트랙트처럼 `for loop 길이`, `storage write 횟수`, `opcode 수준 비용`까지 설계 단계에서 민감하게 다루는 구조와는 다르다.

#### 5.2 저장공간(Storage)

은행 시스템의 저장공간은 기본적으로 운영 가능한 데이터베이스다. 스키마를 변경할 수 있고, 컬럼을 추가하거나 인덱스를 설계할 수 있으며, 운영자가 SQL이나 프로그램을 통해 보정할 수 있다. 저장 구조는 서비스 요구사항에 따라 진화할 수 있다.

반면 스마트컨트랙트의 storage는 체인 전체가 합의하는 영구 상태다. 단순히 “데이터를 보관하는 DB”가 아니라, 모든 노드가 복제하고 검증해야 하는 고비용 상태 공간이다. 따라서 작은 값 하나를 저장하는 것조차 비용과 구조 안정성을 함께 고려해야 한다. 특히 업그레이드 가능한 컨트랙트에서는 storage layout이 깨지면 기존 상태 해석이 달라질 수 있으므로, 은행 DB보다 훨씬 더 보수적으로 설계해야 한다.

즉 은행 DB는 “운영 가능한 저장소”이고, 컨트랙트 storage는 “가급적 적게 써야 하는 공용 영구 상태”에 가깝다.

#### 5.3 함수 구조

은행의 송금 서비스 함수는 일반적으로 단일 함수 호출이 아니라 하나의 업무 프로세스다. 예를 들어 송금 서비스라 하더라도 내부적으로는 입력 검증, 권한 체크, 한도 확인, 계좌 조회, 수수료 계산, 장부 반영, 대외 전문 송신, 실패 시 rollback 또는 재처리, 로그 저장, 고객 통지까지 하나의 서비스 흐름으로 묶여 있다. 함수는 이 전체를 orchestration 하는 도구다.

반면 스마트컨트랙트 함수는 공개된 상태 전이 규칙이다. 외부 사용자가 함수를 호출하면 조건에 맞는 경우 상태가 변경되고, 맞지 않으면 revert 된다. 실행 결과는 체인에 직접 반영되므로, 함수 하나가 곧 시스템 상태 변화를 의미한다. 따라서 컨트랙트 함수는 은행 함수보다 훨씬 짧고, 역할이 명확하며, storage 접근과 외부 호출을 최소화하도록 설계된다.

이를 정리하면 다음과 같다.

| 항목 | 은행 시스템 | 스마트컨트랙트 |
|---|---|---|
| 비용 모델 | 인프라 비용 중심 | 연산 단위별 gas 비용 |
| 저장공간 | 운영 가능한 DB | 합의된 영구 상태 |
| 구조 변경 | 비교적 용이 | 매우 보수적 필요 |
| 함수 역할 | 업무 프로세스 orchestration | 공개 상태 전이 규칙 |
| 실패 처리 | rollback, 재처리, 운영 보정 | revert, reverse tx |
| 외부 호출 | 신뢰 가능한 내부/기관 연계 | 비신뢰 환경 공개 호출 |

결국 스마트컨트랙트를 은행 애플리케이션처럼 설계하면 비용과 유연성 측면에서 비효율이 커지고, 반대로 은행 시스템을 스마트컨트랙트처럼 단순 상태 전이만으로 보려 하면 금융 서비스의 실무적 요구사항을 놓치게 된다.

## 6. 송금 서비스 함수와 Solidity의 transfer / call을 어떻게 비교해야 하는가

은행의 송금 서비스는 Solidity의 `transfer` 또는 `call`과 1:1로 대응하지 않으며, 굳이 비교한다면 은행의 송금 서비스는 “업무 전체 프로세스”이고 `transfer`나 `call`은 그중 자금 이전 실행에 해당하는 저수준 동작에 가깝다.

은행의 송금 서비스는 단순 송금 명령이 아니다. 사용자가 송금 버튼을 눌렀을 때 시스템 내부에서는 계좌 상태 확인, 비밀번호 또는 인증 확인, 한도 체크, 수수료 산정, 수취 정보 검증, 장부 차감, 대외 연계, 거래 결과 기록, 고객 통지 등이 함께 수행된다. 즉 은행의 송금 서비스 함수는 서비스 오케스트레이션 함수다.

반면 Solidity의 `transfer`나 `call`은 자산을 외부 주소로 보내는 저수준 실행 방식이다. 특히 최근에는 `transfer`보다 `call`을 더 많이 사용한다. `transfer`는 2300 gas만 전달하는 고정 정책을 가지기 때문에, 수신 측 로직이 조금만 복잡해도 실패할 수 있다. 반면 `call`은 더 많은 gas를 전달할 수 있어 유연하지만, 그만큼 재진입 공격(reentrancy) 가능성도 커진다. 따라서 스마트컨트랙트에서는 `call`을 사용하되, 상태를 먼저 갱신하고 외부 호출을 나중에 수행하는 Checks-Effects-Interactions 패턴이나 reentrancy guard를 함께 사용한다.

즉 다음처럼 이해하는 것이 적절하다.

- **은행 송금 서비스 함수** = 입력 검증부터 장부 반영, 대외 연계까지 포함하는 전체 프로세스
- **Solidity `call`** = 그 프로세스 안에서 실제 자산을 외부로 보내는 실행 수단 중 하나

따라서 은행 송금과 `transfer/call`을 직접 비교하면 abstraction level이 맞지 않는다. 오히려 비교해야 하는 것은 다음과 같다.

- 은행 송금 서비스 전체 ↔ 스마트컨트랙트 + 오프체인 서비스 조합
- 은행의 실제 지급 실행 ↔ 컨트랙트의 `call` 또는 토큰 `transfer`

이 차이를 구분해야 스마트컨트랙트를 금융 시스템에 어떻게 적용할지 설계를 정확하게 할 수 있다.

##  7. 일부 온체인 시스템 내에서 예금토큰과 스테이블코인을 사용할 때의 프로세스 차이

예금토큰과 스테이블코인은 모두 온체인 결제 수단이 될 수 있지만, 은행 시스템과의 접합성 측면에서는 예금토큰이 더 자연스럽고, 개방형 생태계 활용 측면에서는 스테이블코인이 더 넓다.

예금토큰은 은행 예금의 토큰화된 표현으로 이해할 수 있다. 즉 은행 내부 장부에 존재하는 예금의 일부를 DLT 상에서 이동 가능한 형태로 표현하는 것이다. 이 경우 은행의 계정 구조, 규제 체계, 고객 잔액 관리와 연결되기 쉬우며, 실무적으로도 기존 은행 시스템과 이어 붙이기 쉽다.

반면 스테이블코인은 일반적으로 별도의 발행자가 법정화폐 또는 안전자산을 준비자산으로 보유하고 발행한 토큰이다. 결제 측면에서 편의성이 크고 개방형 네트워크에서 활용도가 높지만, 은행 예금 그 자체와 동일한 청구 구조라고 보기는 어렵다. 따라서 은행의 내부 장부와 1:1로 대응시키려면 추가적인 정산·보증·환매 구조가 필요할 수 있다.

프로세스 관점에서 보면 둘 다 하이브리드 구조에서 다음과 같이 활용될 수 있다.

1. 고객이 은행에 송금을 요청한다.  
2. 은행 내부 시스템이 잔액, 한도, AML, 수취 기관 정보 등을 검증한다.  
3. 거래가 승인되면, 해당 금액만큼 예금토큰 또는 스테이블코인을 준비한다.  
4. 온체인에서 상대 기관 또는 수취 주소로 자산을 이전한다.  
5. 상대 기관은 이를 확인하고 자기 내부 장부에 최종 입금 반영한다.  
6. 오류가 발생하면 오프체인 장부에서는 보정이 가능하지만, 온체인 구간은 필요시 reverse transaction으로 상쇄한다.

이 구조에서 중요한 점은, 예금토큰이나 스테이블코인이 등장하더라도 **금융 서비스 전체가 온체인으로 이동하는 것은 아니라는 점**이다. 오히려 온체인 자산은 기존 금융 장부와 상대 기관 장부를 연결하는 settlement bridge 역할을 수행하게 된다.

## 8. 장부형 시스템의 장점과 블록체인의 장점을 함께 살리는 구조 제안

가장 현실적인 구조는 “오프체인 장부 기반의 서비스 처리”와 “온체인 기반의 최종 가치 이전”을 분리한 이중 구조이며, 이 방식이 은행의 운영 가능성과 블록체인의 투명성·최종성을 함께 살릴 수 있다.

장부형 시스템의 강점은 다음과 같다.

- 정정과 보정이 쉽다.
- 예외 처리와 운영 개입이 가능하다.
- 규제나 정책 변경에 유연하다.
- 장애 복구와 재처리가 가능하다.
- 고객 보호와 책임 분담 구조가 명확하다.

블록체인의 강점은 다음과 같다.

- 최종 결제 기록이 명확하다.
- 여러 기관이 공유 가능한 상태 기록을 가진다.
- 프로그래머블한 자금 이전이 가능하다.
- 일정 조건 하에서 실시간 또는 준실시간 결제가 가능하다.
- 자산과 결제를 같은 플랫폼에서 다룰 수 있다.

이 두 장점을 함께 살리려면 다음과 같은 구조가 적절하다.

### 제안 구조

#### 1) 오프체인 서비스 계층
- 고객 요청 접수
- KYC/AML/한도/정책 검증
- 거래 상태 관리 (pending, approved, failed 등)
- 내부 장부 차감 및 회계 처리
- 고객 통지, 민원, 운영 보정

#### 2) 온체인 settlement 계층
- 승인된 거래만 진입
- 예금토큰 또는 스테이블코인 이동
- 기관 간 최종 가치 이전
- 원거래 참조 가능한 온체인 transaction id 기록

#### 3) 오프체인 사후 정산 및 보정 계층
- 상대 기관 장부 반영
- 온체인 거래 참조
- reverse transaction 또는 adjustment 연결
- 감사 및 규제 보고

이 구조는 순수 온체인 모델보다 복잡해 보일 수 있지만, 실제 금융 서비스 요구사항에는 훨씬 잘 맞는다. 특히 은행 내부 장부의 상태와 온체인 거래 상태를 명확히 분리하고 매핑하는 것이 중요하다. 예를 들어 다음과 같이 상태를 구분할 수 있다.

- `REQUESTED`
- `APPROVED`
- `TOKEN_PREPARED`
- `ONCHAIN_SENT`
- `SETTLED`
- `REVERSED`
- `ADJUSTED`

이렇게 하면 은행의 운영 장부와 블록체인 기록을 연결할 수 있고, 온체인 finality와 오프체인 운영 가능성을 동시에 확보할 수 있다.
